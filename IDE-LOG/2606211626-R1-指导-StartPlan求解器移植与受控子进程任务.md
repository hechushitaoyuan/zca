## 4. 实现要求

### 4.1 Solver

1. 以 PR #5 提交 `525ff04`/头提交 `aac09c7` 的 `captcha_node/solver.js` 为技术基线，不使用当前上游中已知 exit code 4 的简化实现。
2. 默认 scene 必须为 `11xygtvd`；region、prefix 从参数传入，并保留安全默认值。
3. 成功只通过约定的 `VERIFY_PARAM=` 输出返回；不得在普通日志打印完整 verifyParam。
4. 失败使用可区分退出码；stderr 只能包含脱敏、限长诊断。
5. 不增加 Chromium/Playwright 等重型依赖，继续使用 Node + jsdom。

### 4.2 Python 受控子进程

1. 子进程必须有可配置超时；超时后终止并完整回收，不能留下僵尸进程。
2. 检查退出码、输出格式和最小 verifyParam 长度；空输出、非零退出、格式错误分别形成可测试异常。
3. stderr/stdout 诊断必须限长并脱敏，不得包含完整 JWT、API Key、verifyParam。
4. 45 秒左右 TTL 缓存与 single-flight 必须可测试：并发请求只启动一次 solver。
5. `invalidate()` 后下一次必须重新求解。
6. 主动配置接口返回的 region 必须随求解结果传入 `X-Aliyun-Captcha-Verify-Region`，不能永远硬编码 `sgp`。
7. 上游 captcha 失败只允许一次窄范围重新求解；3012 不触发 captcha 重试。

### 4.3 ARM64 与镜像

1. 不引入仅支持 amd64 的原生依赖。
2. Dockerfile 必须保持 `linux/amd64`、`linux/arm64` 可构建的基础条件。
3. 如果当前 Dockerfile 已满足要求，不要为“看起来更完整”而改动。

## 5. 测试要求（默认全部 mock）

至少覆盖：

1. TTL 内重复获取只启动一次子进程。
2. 多个并发调用 single-flight，只启动一次子进程。
3. `invalidate()` 后重新启动。
4. 动态 scene/region/prefix 正确传给 Node。
5. region 正确进入最终请求头。
6. timeout 会 kill 并回收子进程。
7. 非零退出码、无标记输出、过短参数均失败且错误信息不泄露完整参数。
8. captcha 3007/403 只重试一次；3012 直接进入冷冻且不调用 invalidate/solver 重试。
9. 原有 `stream:false` 和成功 SSE 不预读测试继续通过。

建议自测命令：

```powershell
python -m compileall -q app tests main.py
python -m unittest discover -s tests -v
node --check captcha_node/solver.js
npm --prefix captcha_node test --if-present
```

不得用真实账号/JWT做测试，不得调用真实 Messages 上游。
