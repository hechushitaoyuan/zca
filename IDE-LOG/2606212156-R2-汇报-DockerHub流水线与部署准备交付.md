### 2.3 `.github/workflows/docker-build.yml`（重写）

| 维度 | 实现 |
|---|---|
| 触发 | `push: [main]` + `tags: v*` + `pull_request: [main]` + `workflow_dispatch`（已从错误的 `master` 改为 `main`） |

## 6. 残余风险

1. **P2｜多架构 build 未本地验证**：arm64 交叉构建依赖 CI 的 QEMU，本机无 Docker 无法预跑。首次 CI 运行若 arm64 层因原生依赖失败需回看——但 R1 已确认 captcha_node 无 amd64-only 原生依赖，jsdom 纯 JS，风险低。
2. **P2｜HEALTHCHECK 真实行为待 CI/VPS 确认**：python urllib 探活在容器内的实际表现（启动期 `start-period=20s` 是否足够 uvicorn 就绪）需 E4 观察，必要时调 `start-period`。
3. **P2｜Secrets 未配置则推送步骤会失败**：CI 的 Docker Hub 登录依赖乙方在仓库配置 `DOCKERHUB_USERNAME`/`DOCKERHUB_TOKEN`；未配置时 push 事件的 build job 会在登录步骤失败（PR 不受影响，因 PR 不登录）。这是预期行为，非缺陷。

### 逐项

- **P1-1 数据目录绕过挂载卷**：`.env.example` 原 `ZCODE_DATA_DIR=./data` 会经 `env_file` 覆盖镜像默认 `/data`，使 SQLite 写入 `/app/data` 而非挂载卷，重建容器丢账号。改为 `/data` 并加注释说明容器部署必须与挂载卷一致。
- **P1-2 端口暴露过大**：`compose.yaml` 端口改为仅绑回环 `127.0.0.1:8047:3000`，实验服务不直接公开；`DEPLOYMENT.md` 同步说明本机 Tunnel/SSH 验证不受影响，公网直连需另行放开并先配网关鉴权。
- **P1-3 外部 PR 无 Secrets 构建失败**：`metadata-action` 的 `images` 原始终依赖 `secrets.DOCKERHUB_USERNAME`，fork PR 拿不到 Secrets 会退化成无效的 `/zca`。改用表达式 `github.event_name == 'pull_request' && 'local/zca' || format('{0}/zca', secrets.DOCKERHUB_USERNAME)`，PR 用非敏感占位名，非 PR 用真实用户名。未写入明文用户名/token。
