## 2. P1-1：容器数据目录绕过持久化卷

`compose.yaml` 挂载 `./data:/data`，但 `.env.example` 设置 `ZCODE_DATA_DIR=./data`。部署时 `env_file: .env` 会覆盖 Dockerfile 的 `/data`，应用实际写入 `/app/data`，SQLite 不进入挂载卷，重建容器可能丢失账号和配置。

**定向修复**：把 `.env.example` 的 `ZCODE_DATA_DIR=./data` 改为 `ZCODE_DATA_DIR=/data`。

## 3. P1-2：实验端口暴露范围过大

当前 `"8047:3000"` 绑定所有宿主机网卡；新库首次启动的网关密钥尚未由部署流程强制初始化。为避免实验服务意外公开，且不影响本机 Cloudflare Tunnel/SSH 验证，应先只监听回环地址。

**定向修复**：把 `compose.yaml` 端口改为 `"127.0.0.1:8047:3000"`，并同步 `docs/DEPLOYMENT.md` 的说明。后续若甲方明确要公网直连，再单独放开并先配置网关鉴权。

## 4. P1-3：外部 PR 无 Secrets 时构建验证会失效

工作流声称 PR 只验证不推送，但 `docker/metadata-action` 的 `images` 始终依赖 `secrets.DOCKERHUB_USERNAME`。来自 fork 的 PR 不会获得仓库 Secrets，镜像名可能变成无效的 `/zca`，使“PR 构建验证”失败。

**定向修复**：PR 事件使用非敏感占位镜像名（例如 `local/zca`）；仅非 PR 事件使用 `${{ secrets.DOCKERHUB_USERNAME }}/zca`。不得把真实用户名或 token 写入 workflow。
