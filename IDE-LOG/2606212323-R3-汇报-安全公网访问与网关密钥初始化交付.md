## 2. 逐文件改动目的

### 2.1 `app/settings.py`

新增 `GATEWAY_KEY = (os.getenv("ZCODE_GATEWAY_KEY", "") or "").strip()`，默认空。仅用于库初始化/补空，注释标明公网监听前必须设强值。未改其他配置。

### 2.2 `app/store.py`（核心安全语义，仅 `_init_db`）

原逻辑：`INSERT OR IGNORE ... ('gateway_key', '')`——新库插空串，既有库忽略。

改为两步：

1. `INSERT OR IGNORE ... ('gateway_key', ?)` 绑定 `settings.GATEWAY_KEY`——**新库**用环境值（未配置则空）初始化。
2. `if settings.GATEWAY_KEY:` 守护下执行 `UPDATE {_META} SET value = ? WHERE key = 'gateway_key' AND value = ''`——**既有库**仅当当前为空且环境值非空时补齐。

三重保护确保"只补空、绝不覆盖"：
- `WHERE value = ''`：只命中空值行，用户已设的非空值不在更新范围。
- `if settings.GATEWAY_KEY`：环境值为空时整段 UPDATE 跳过，保持现有兼容行为。
- `INSERT OR IGNORE`：既有库的 INSERT 被忽略，不影响已有行。

### 2.3 `compose.yaml`

端口 `"127.0.0.1:8047:3000"` → `"${ZCA_BIND_IP:-127.0.0.1}:8047:3000"`。默认仍回环安全值；注释说明公网放开的前置条件。

### 2.4 `.env.example`

- 服务段新增 `ZCA_BIND_IP=127.0.0.1`，解释 `127.0.0.1`（仅本机）vs `0.0.0.0`（公网）差异与放开前置。
- 鉴权段新增 `ZCODE_GATEWAY_KEY=`（默认空），说明新库初始化 / 既有库补空 / 不覆盖非空三种行为。

### 2.5 `docs/DEPLOYMENT.md`

更新端口说明为 `ZCA_BIND_IP` 机制；新增「3.1 公网直连」节，明确放开前必须：①非空强 `ZCODE_ADMIN_KEY`；②非空强随机 `ZCODE_GATEWAY_KEY`（否则网关无鉴权属高风险）；③云防火墙仅放行预期来源、未误开 8046。含回退步骤与「不回显真实 Key」提醒。

## 5. 残余风险

1. **P1｜公网放开依赖人工正确配置**：`ZCA_BIND_IP=0.0.0.0` 的安全性完全取决于部署者是否同时设了强 `ADMIN_KEY`/`GATEWAY_KEY` 并收敛防火墙。代码层做了「空网关 Key 自动补齐」，但**不能强制**部署者用强值，也无法在放开公网前阻断空 Key 启动。文档已重点警示；如需更强约束（如 `0.0.0.0` 且网关 Key 为空时拒绝启动），属新需求，需乙方评估后另发任务。
2. **P2｜既有库补空为一次性**：环境值仅在启动时补齐空值。若用户启动后又在后台把网关 Key 清空，环境值不会再次补齐（需重启）。符合「以数据库为准」的现有语义，登记备查。
3. **P2｜真实补齐行为未经 E3 验证**：临时库测试已覆盖四类语义，但真实 `data/accounts.db`（可能含历史 meta 行）的补齐仅推断为同逻辑，未在真实库实测。
