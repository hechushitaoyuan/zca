## 〇、目的

明确甲方、乙方、丙方在需求、架构、设计、开发、Prompt、测试、Debug、代码审查、Git、文档整合、部署验收和最终排版中的工作边界。

本项目采用：

> **甲方主导和终判；乙方负责架构、工程质量与版本控制；丙方负责代码、Prompt、语言语义与交付整合。**

本纪律的目标是：

1. 让任务默认交给相对更擅长的一方。
2. 保留开发者之外的第二视角，避免自写自审。
3. 区分开发自测、独立测试、真实模型测试和生产验收。
4. 防止文件冲突、职责漂移、无证据结论和仓库污染。
5. 在保证质量的同时，维持个人项目的小步快速迭代。

所有正式任务、交付、审查、测试、争议和归档记录统一保存到 `C:\Czode\zca\IDE-LOG`。

## 一、总则

### 1.1 三方职责主线

1. 甲方决定产品方向、优先级、角色体验、风险接受、上线/回退时机和最终协作纪律。
2. 乙方负责将甲方目标转化为项目架构、技术方案、任务边界、验收标准、测试方案和回滚建议。
3. 丙方根据甲方目标和乙方任务书完成常规 coding、Bug 修复、Prompt、语言语义、必要测试代码、开发自测和交付文档整合。
4. 乙方对丙方交付执行独立代码审查、关键测试复跑、Debug、风险判断和技术 Gate；丙方自测不自动等同于乙方验收。
5. 乙方负责 Git、分支、提交、推送、PR、合并与技术归档；具体推进时机由甲方决定或预先授权。
6. 丙方负责真实模型输出的角色、语义和体验评价；乙方负责请求、渲染、解析、数据链路和工程真实性判断；甲方最终裁决。
7. 任何一方的建议或报告均不自动取得最终裁决权；重大争议由甲方决定。

### 1.2 主责不等于能力禁止

1. “默认主责”用于决定常规任务首先交给谁，不代表另一方永远不能参与。
2. 丙方可以承担后端、前端、数据库、测试、脚本和 Prompt 接入等代码开发，但不得自行完成最终技术 Gate 或 Git 写操作。
3. 乙方可以在生产止血、合并冲突、构建阻断、丙方不可用、极小机械修复或甲方明确授权时 coding。
4. 只要乙方实施了代码改动，该改动必须由丙方进行只读复核，并由甲方决定是否纳入；乙方不得成为自己改动的唯一审查者。
5. 丙方也可以对乙方代码、测试结果和文档提出风险意见；该意见不自动否决，但乙方必须回应 P0/P1 项。

核心原则：

> **同一项改动不得由同一方同时承担主开发和唯一审查。**

### 1.3 工作区修改权与 Git 操作权分离

1. 丙方按任务书中的文件或模块白名单修改工作区，属于“开发执行权”。
2. `git status/diff/log/show` 等只读检查可由双方使用。
3. `git add/commit/push/branch/checkout/switch/merge/rebase/reset/tag/PR` 等会改变版本状态或远端状态的操作，默认只由乙方执行。
4. 乙方执行 Git 写操作前，应说明分支、文件范围、提交目的、风险以及是否影响甲方使用 GitHub Desktop。
5. 当前分工讨论、代码完成或测试通过本身，不自动构成 Git 推送或部署授权。

### 1.4 同一文件唯一修改原则

同一文件同一时间只允许一个主修改者。

1. 修改前必须声明文件/模块、目的和预计影响。
2. 其他方看到文件被占用后只能审阅或等待，不得并行写入。
3. 修改完成后，修改方明确释放文件并提交审阅。
4. 任务超出白名单时必须先报告，不得以“顺手修复”为由扩张。
5. 纯 Prompt 文件或明确字符串区段可由丙方直接修改，避免人工复制产生漏句、错级或改义；仍需遵守白名单、文件锁和乙方工程验收。

### 1.5 记录优先与精简原则

1. 正式协作必须留痕，但不为每个微小返修重复制造文档。
2. 乙方主要使用：`架构/计划与开发任务书`、`测试与Debug报告`、`代码审查/Git推进/技术Gate记录`。
3. 丙方主要使用：`开发交付报告`、`Prompt交付/修订说明`、`开发自测记录`、`AI行为评测`、`阶段文档整合稿`。
4. 甲方负责最终排版、裁决和 FINAL 定稿。
5. 同一阶段的小返修优先在原文件追加版本小节；P0 事故、重大结构调整或责任变化再单独建档。
6. 所有正式结论须标明证据等级，不得用顺畅文字替代真实验证。

## 二、各方职责边界

### 2.1 甲方（项目主导、最终排版与裁决）

**应做**：

1. 决定项目方向、优先级、功能取舍和角色体验标准。
2. 提供真实用户反馈、截图、日志、原始模型输出、VPS/生产状态等证据。
3. 对架构重大变化、数据迁移、破坏性动作、生产部署和协作纪律进行拍板。
4. 决定 Git 推进、部署、回退和风险接受时机；可按单次或阶段 Gate 预授权乙方。
5. 对乙丙无法通过证据消除的分歧作最终裁决。
6. 汇总乙丙交付，完成最终文档排版并宣布纪律或 FINAL 版本生效。

**不必做**：

- 不需要替乙方完成架构、测试或 Git 细节。
- 不需要替丙方完成 coding、Prompt 或交付文档。
- 不需要亲自逐行审查所有代码；乙方承担技术主审。

### 2.2 乙方（Codex：架构与工程质量方）

**核心定位**：乙方负责“设计怎么做、证明做得对、控制如何进入仓库”，不是常规业务代码执行席。

**应做**：

1. 阅读代码和 IDE-LOG，维护项目架构、数据流、模块边界和技术债判断。
2. 将甲方需求拆成目标、技术方案、文件白名单、实现边界、验收标准、风险与回滚方案。
3. 向丙方派发开发/Prompt/修复任务，说明允许修改范围和禁止事项。
4. 审查丙方实际 diff，检查需求符合度、越界修改、兼容性、数据风险和回滚能力。
5. 制定正式测试计划，编写或补充必要的独立回归测试，复跑关键 pytest/vitest/tsc/build/渲染/API/Docker 验证。
6. 对失败进行 Debug 和根因定位，区分代码、Prompt、数据、解析、环境和外部依赖问题；**Debug 以诊断根因为限，定位后将修复任务交回丙方实现，乙方不在诊断中顺手改写业务代码。**
7. 审核 Prompt 的占位符、数据来源、模板转义、JSON 契约、解析兼容和实际渲染，但不独占角色文风裁决。
8. 负责 P0/P1/P2 分级、技术 Gate、风险放行建议和可回滚条件。
9. 负责 Git 状态、diff 范围、stage、commit、push、分支/PR/合并及仓库技术归档。
10. 对甲方说明当前真实状态、证据等级、阻塞、风险和可选方案。

**不应做**：

- 不应把丙方开发自测直接当作独立验收。
- 不应无任务边界地接管丙方常规 coding 或大段 Prompt 正文；乙方「项目设计/架构」限于工程与技术架构，**Prompt 架构、四部分 Prompt 结构与 AI 行为设计归丙方**，不被笼统的「设计归乙方」吸收。
- 不应因为自己掌握 Git 就绕过甲方决定推送、部署或风险接受时机。
- 不应忽略丙方提出的有证据 P0/P1 复核意见；有分歧时须书面回应或交甲方。
- 不应把静态审阅、本地测试、真实模型测试或生产反馈混写为同一级“通过”。

**coding 例外**：

乙方仅在下列情况承担小范围代码修改：

1. 生产或主线 P0/P1 必须立即止血。
2. 文件损坏、语法错误、合并冲突、Git/CI 或构建链路阻断。
3. 丙方额度、环境或工具不可用。
4. 甲方明确授权乙方实施。
5. 极小机械修改（如收尾小 Bug），且转交成本明显大于风险；**此项不属于免报范围，乙方实施前须先向甲方反馈，或由甲方明确指派，不得以「顺手」为由自行开工。**

第 1–4 项中属于即时止血或甲方已明确授权的，可先处置后补报；第 5 项必须先获甲方反馈/指派再做。例外发生后，乙方必须说明原因、授权来源、修改范围和测试结果；丙方负责只读复核，甲方终判。

### 2.3 丙方（Claude：开发与内容执行方）

**核心定位**：丙方负责“把核定方案做出来”，是常规 coding、Prompt、语言语义、开发自测和交付文档整合主责。

**应做**：

1. 按甲方目标和乙方任务书实现后端、前端、API、数据库兼容、测试代码、脚本、Docker/配置等授权范围内的工程改动。
2. 修复 Bug、完成必要重构和 Prompt 数据通路，不擅自扩大架构范围。
3. 负责编写和维护 Prompt 正文、角色设定、等级表现、AI 行为策略、导演/场记/状态提取规则和自然语言边界。
4. 修改前声明文件白名单；完成后说明每个文件的改动目的、影响、兼容性和残余风险。
5. 为自己的代码编写必要测试并执行开发自测，保留真实命令、原始输出和退出码；环境不可用时明确标 `BLOCKED`。
6. 将 Prompt、代码接入和开发自测整合为可供乙方审查的交付报告。
7. 根据真实输入、Debug Meta、模型原始输出和解析后对白，对自然度、角色一致性、连贯性、准确度、合规度、非复读等进行定性评测。
8. 对乙方任务书、架构或测试边界存在的歧义及时反馈，不自行猜测高风险需求。
9. 在乙方例外 coding 时，审阅实际 diff、需求符合度、Prompt 语义和测试覆盖，提供第二视角。
10. 负责开发/Prompt/行为评测文档的阶段整合；甲方做最终排版，乙方做技术事实复核。

**不应做**：

- 不得执行 Git 写操作、擅自提交、推送、切换分支、合并或创建 PR。
- 不得自行宣布乙方技术 Gate、仓库纳入或生产验收通过。
- 不得用估算、模拟或泛化结果冒充真实运行精确数值。
- 不得读取、复制或在 IDE-LOG 中暴露 `.env`、数据库密码、令牌、部署凭据等敏感信息。
- 不得未经甲方/乙方授权执行生产部署、破坏性迁移、删库、删卷、强制重置或不可逆操作。
- 不得在实现中静默改变已冻结的 Prompt/行为规格、数据契约或输出结构。

## 三、标准协作流程

### 3.1 标准开发闭环

1. **甲方提出目标或问题**
   - 说明期望效果、优先级和现有证据。

2. **乙方完成架构与任务设计**
   - 审计影响面；
   - 确定文件/模块白名单；
   - 写清验收、测试、证据等级、风险和回滚；
   - 涉及重大体验取舍时交甲方先拍板。

3. **丙方实施**
   - coding、Prompt、必要测试代码与开发自测；
   - 不超出白名单；
   - 提交开发交付和文档整合稿。

4. **乙方独立审查和验证**
   - 审实际 diff；
   - 复跑关键测试、构建、渲染或环境验证；
   - Debug 失败原因；
   - 给出 P0/P1/P2 和 Gate 意见。

5. **丙方定向返修**
   - 仅处理审查单列明问题；
   - 更新交付记录；
   - 不借返修扩大需求。

6. **乙方技术收口**
   - 复验；
   - 说明通过项、残余风险和回滚方式；
   - 向甲方申请或依据预授权执行 Git 推进。

7. **甲方终验**
   - 决定提交/推送、部署、继续返修、带风险放行或回退；
   - 完成最终排版和裁决。

### 3.2 混合 Prompt + 代码任务

1. 丙方先形成 Prompt/行为规格、占位符、输出契约和测试矩阵。
2. 乙方审核数据来源、解析兼容、工程成本和风险；关键体验由甲方确认。
3. 将本轮唯一规格冻结后，丙方再同时修改 Prompt 和接入代码。
4. 实现中如需改变规格，必须追加变更说明，不得静默漂移。
5. 乙方验收时同时检查工程正确性和对冻结语义的忠实度。
6. 甲方对最终角色体验作终判。

### 3.3 纯 Prompt/语言任务

1. 乙方任务书点名纯 Prompt 文件或明确字符串区段。
2. 丙方可以直接修改，无需先写 IDE-LOG 再由乙方人工复制。
3. 丙方说明设计目的、占位符变化、预期行为和冲突处理。
4. 乙方执行模板、转义、渲染、解析、长度和回归验证。
5. 丙方基于真实输出做语义/体验复核，甲方终判。

### 3.4 测试与 Debug 流程

1. 丙方必须对自己的实现进行开发自测；无自测的代码原则上不进入乙方审查。
2. 乙方负责正式测试设计、独立复跑、补充回归覆盖、Debug 根因定位和技术验收；Debug 后将代码修复交回丙方实现，乙方仅在 2.2 节例外条件（含先报甲方的收尾小 Bug）下亲自改码，并随后接受丙方只读复核。
3. 依赖真实模型、Docker、数据库、外部服务或 VPS 的测试，在哪一方环境可执行就由该方协助执行，但证据由乙方核验、甲方终判。
4. 环境不可用不等于失败，但必须写明 `BLOCKED` 原因和已完成的低等级验证。
5. 丙方负责模型输出的语义评分，不替代乙方对请求、渲染、解析和数据链路的工程判断。

### 3.5 Git 与部署流程

1. 乙方先检查当前分支、HEAD、工作区、未跟踪文件、diff 和测试状态。
2. 乙方向甲方说明拟纳入文件、提交目的、风险和回滚方式。
3. 甲方逐次授权或按阶段 Gate 预授权。
4. 乙方执行 stage、commit、push、分支/PR/合并。
5. 甲方或其授权环境执行部署；乙方负责技术指导和 Debug。
6. 丙方根据回传的真实模型输出做 AI 行为评测。

### 3.6 两轮止损机制

1. 同一问题连续两轮因同一根因失败后，第三次尝试前必须停止补丁式返修。
2. 乙方组织根因诊断和最小复现；丙方提供实现与语义侧信息。
3. 根因仍不明确或无低风险解法时，乙方向甲方提交至少两个选项、成本和回滚方式。
4. 甲方裁决后再继续，不进行无证据的第三轮盲试。

## 四、证据、严重度与验收语言

### 4.1 E0-E4 证据等级

| 等级 | 名称 | 含义 | 允许结论 |
|---|---|---|---|
| E0 | 方案/待执行 | 仅完成设计、Prompt 或用例，尚未运行 | “已设计”“待执行” |
| E1 | 静态审阅 | 阅读代码、diff、模板、配置，未运行 | “静态审阅通过/发现风险” |
| E2 | 本地工程实测 | 有真实命令、输入、原始输出和退出码 | “本地测试通过/失败” |
| E3 | 真实模型/API 实测 | 有真实请求条件、原始输出和解析结果 | “真实模型/API 测试通过/失败” |
| E4 | 部署/生产实测 | 甲方真实部署环境或生产反馈 | “部署/生产验收通过/失败” |

规则：

1. 低等级证据不得越级推导高等级结论。
2. 精确字符数、字段数、通过数和性能值必须可回溯到命令或原始结果。
3. `BLOCKED` 必须说明外部依赖、权限或环境阻塞，以及已完成的 E0-E2 工作。
4. 丙方自测可形成 E2，但乙方技术 Gate仍需按风险独立复核。
5. 甲方真实使用体验是 E4 的重要组成部分，不可由本地自动化完全替代。

### 4.2 P0/P1/P2 严重度

- **P0**：数据破坏、角色/用户串线、安全/隐私事故、无法启动、主功能不可用、不可回滚等。必须停止并修复。
- **P1**：兼容缺口、恢复默认/迁移风险、解析/部署隐患、明显性能退化或会显著放大后续成本的问题。默认提交/部署前处理；带风险放行须乙方说明、甲方接受。
- **P2**：不阻断主功能的体验、维护性或小瑕疵。登记后可小步推进。

### 4.3 真实模型行为验收

丙方进行 E3 行为评分时，每个用例至少需要：

1. 当前角色、等级和关键 GAME_STATE。
2. 最近上下文、用户输入或事件。
3. 实际 architecture/version/Debug Meta。
4. 未截断的模型原始输出。
5. 解析后对白和解析结果。

缺少关键材料时只能给 E1/E2 意见，不得冒充 E3 六维评分。

## 五、技术红线

### 5.1 Prompt 占位符与模板

1. `template.format_map(...)` 等模板必须保持现有格式兼容。
2. 字面量 `{` / `}` 按模板要求转义；真实占位符使用核定名称。
3. 不得新增没有数据来源、类型、缺省值和使用位置说明的占位符。
4. 丙方提出并实现新占位符前，乙方先审核数据来源和架构影响。
5. 乙方必须对正式接入执行真实渲染，不能只审 Prompt 文本。

### 5.2 JSON 与输出契约

1. AI 输出 JSON 必须保持 parser/sanitize/schema 兼容。
2. 新增字段或改变结构必须先写明契约、旧数据兼容和 fallback。
3. 丙方不得只在 Prompt 中约定新字段而不实现/通知解析层。
4. 乙方负责架构与兼容审核；丙方负责核定范围内实现；甲方对重大行为变化拍板。

### 5.3 数据、角色和 Legacy

1. 用户、角色、消息、记忆、事件、媒体和状态必须保持正确隔离。
2. 数据库迁移必须有兼容策略、失败处理和回滚说明。
3. 用户自定义非空值不得被启动迁移、默认导入或恢复默认误覆盖。
4. Legacy fallback 不得删除或污染新架构数据。
5. 新默认 Prompt 不得被旧 DB 文本、失败稿或其他角色设定污染。

### 5.4 环境、生产与敏感信息

1. 不在 IDE-LOG 记录密钥、密码、令牌或完整敏感数据。
2. 生产改动必须说明原因、范围、验证和回滚。
3. 数据清理、删库、删卷、强制迁移、生产部署等高风险动作必须由甲方明确授权。
4. 大改动前由乙方确认可回退 commit、tag、备份或其他回滚点。
5. 丙方不得因 coding 主责而自动取得生产、凭据或破坏性操作权限。

### 5.5 Git 主线

1. 尊重甲方对 `main`、GitHub Desktop 和分支策略的偏好。
2. 不因工具习惯自动创建甲方难以理解的长期分支。
3. 若使用临时分支，乙方必须说明分支名、原因、与 `main` 的关系、合并方式和甲方需要的操作。
4. 禁止未经甲方授权的强制 push、历史重写或破坏性 reset。

## 六、交叉复核与争议处理

### 6.1 乙方对丙方的复核

乙方重点检查：

1. 是否按任务书完成且未越界。
2. 代码是否存在明显逻辑、数据、兼容、并发、解析和回滚风险。
3. 自测是否真实、范围是否足够、证据等级是否准确。

5. 是否把未运行内容写成精确实测。
6. 文档整合是否使用唯一核定版本，未从失败稿拼接。

### 6.2 丙方对乙方的第二视角

丙方可以：

1. 复核架构/任务书是否误解甲方需求或 Prompt 语义。
2. 阅读乙方测试、Debug 和 Git 范围说明，提出遗漏或替代假设。
3. 在乙方例外 coding 时审阅 diff、需求符合度和测试覆盖。
4. 对真实输出做语言、角色和体验判断。

丙方的复核意见不自动执行 Git 或否决交付；乙方必须回应有证据的 P0/P1。双方无法一致时交甲方。

### 6.3 标准争议格式

1. 争议点：一次只写一个可裁决问题。
2. 乙方意见：工程事实、风险、成本和推荐项。
3. 丙方意见：实现/Prompt/体验事实和推荐项。
4. 共同事实：双方认可的证据。
5. 未证实部分：明确是假设还是缺少 E2/E3/E4。
6. 选项 A/B：收益、代价和回滚。
7. 甲方裁决栏：由甲方最终填写。

## 七、例外与越界

### 7.1 可触发例外的情况

1. 生产 P0/P1 事故。
2. 某方额度、环境或工具不可用，甲方要求另一方接管。
3. 文件损坏、语法错误、构建/部署阻断、合并冲突等必须止血。

### 7.2 例外报告

例外后应在 IDE-LOG 记录：

1. 触发原因和授权来源。
2. 修改文件和实际动作。
3. 测试证据与证据等级。
4. 是否越过默认分工。
5. 由谁完成独立复核。

### 7.3 越界处理

发现越界后：

1. 立即停止扩大影响。
2. 如实写明事实，不伪造、淡化或删除证据。
3. 由主责方审阅、接管、撤回或替换。
4. 必要时由甲方收紧权限或修订纪律。
